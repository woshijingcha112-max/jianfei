# spec

## 硬命令

1. 未被“当前软规范”允许的内容，不得实现。
2. 属于后续阶段且当前任务不直接依赖的代码、接口、页面、结构，不得提前创建。
3. 接口契约、数据库语义、返回层级、状态模型等高影响决策未明确时，不得自行定死，必须先确认。
4. 占位、骨架、路由、空返回、空实现，不得算业务完成。
5. `减肥计划拍照识别软件开发设计文档.md` 为只读基线，禁止修改。
6. 新增通用组件前，如有必要，优先评估成熟且可落地的开源组件方案；确认不合适后再手写。
7. 本次执行若偏离“当前软规范”，必须先写 `project_develop_log.md`，再决定是否更新本文件。

## 当前软规范

- 当前阶段：核心功能真实化
- 当前只允许：
  - 在首页、拍照、识别结果、目标页现有结构内，补齐“拍照 -> 上传 -> 压缩判定 -> AI 识别 -> 结果展示 -> 保存”的最小真实主链路
  - 启用单数据源与 MyBatis-Plus，重写当前主链路所需最小表基线
  - 对接 `/goal/get`、`/goal/save`、`/diet/photo/upload`、`/diet/record/save`、`/diet/record/list`、`/diet/stat/today`
  - 安卓端以真实 repository、真实拍照入口与真实上传结果替换核心主链路中的 mock 数据
- 当前重点：
  - 首页总览真值固定来自 `/diet/stat/today`
  - 拍照页允许接入真实拍照入口并获取拍照结果，但只做当前主链路直接需要的最小实现
  - 图片上传后由后端按“非必要不压缩”的原则处理，再调用真实视觉模型识别
  - 识别提示词策略本轮不展开优化，固定提示词直接写入后端调用参数
  - 识别结果接口契约要继续为后续人工修正留出空间
  - `BO` 仅在存在复杂内部聚合时允许新增，默认只用 `PO / DTO / VO`
- 当前禁止：
  - 扩展搜索页、`food/search`、`food/detail`
  - 实现 `diet/record/delete`、`weight/record`、`period/*`
  - 实现菜单库/索引库更深层检索、人工修正增强交互
  - 超出首页主链路直接需要的结构性扩张
- 当前完成：首页、拍照、识别结果、目标页的现有结构与首页优先 design language 已落地，可作为核心功能真实化改造基座
- 当前未完成：
  - 菜单库/索引库更深层检索与搜索页扩展
  - 人工修正增强交互
  - 删除记录、体重记录、生理期相关能力
