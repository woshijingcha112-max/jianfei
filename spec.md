# spec

## 硬边界

1. 只允许实现当前任务直接需要的最小改动；不属于当前任务直接依赖的代码、接口、页面、结构，不得提前创建。
2. 接口契约、数据库语义、返回层级、状态模型等高影响决策未明确时，不得自行定案，必须先确认。
3. 占位、骨架、路由、空返回、空实现，不得算业务完成。
4. `减肥计划拍照识别软件开发设计文档.md` 为只读基线，禁止修改。
5. `feature_status.md` 是项目完成度真值；项目进度、已完成/未完成、阻塞项不得写入本文件冒充规范。
6. `project_develop_log.md` 记录阶段性策略、偏差、收口计划与复盘；本文件只记录稳定约束与工程软规范。

## 软规范

### 适用原则

- 软规范用于约束项目实现风格、工程写法、抽象复用和稳定协作习惯，不用于体现项目进度。
- 软规范与硬边界冲突时，以硬边界为准。
- 软规范保持简洁，只保留跨迭代仍成立的稳定规则。

### 实现原则

- 实现时优先复用已有工具类、基础封装、通用 UI 组件；确认现有抽象不适配后，才允许新增重复实现。
- 新增通用能力前，优先评估成熟且可落地的开源方案；确认不合适后再手写。
- 默认坚持最小改动原则，避免为了“未来可能会用”提前铺设大面积结构。
- 后端领域对象默认使用 `PO / DTO / VO`；仅在确有复杂内部聚合时允许新增 `BO`。
- 前后端接口与状态结构在满足当前任务的前提下，应为后续扩展保留必要空间，但禁止空泛预埋无实际落点的大结构。

### 代码风格与注释日志

- 中文是默认注释与业务日志语言；对外协议字段名是否使用英文，以现有技术栈和接口契约为准。
- 每个 Controller 必须有中文类注释；公开接口方法必须有中文方法注释。
- 每个 Service 接口必须有中文类注释；接口方法必须有中文方法注释。
- Service 中复杂逻辑必须按逻辑块分段，段前使用独立单行中文注释；禁止行内注释。
- 注释应解释职责、边界和意图，不得用注释重复代码表面含义。
- 后端日志默认使用 `Slf4j` 与统一日志配置；日志应优先落在入口、关键分支、外部调用、异常和结果汇总位置。
- 安卓日志遵循同样原则，优先记录状态切换、关键交互和外部依赖调用，不添加噪声日志。

### AI 与外部能力接入

- 多模型或多平台接入时，应尽量把平台差异收敛在少数适配点，避免平台细节散落到业务流程中；不为暂时不会发生的扩展提前做过度设计。
- 平台凭证、接口地址、模型名、开关等外部依赖参数统一走配置，不在业务代码中硬编码。
- 提示词、结构化字段映射、兜底策略等易变内容应集中管理，避免散落在多个业务方法中。
